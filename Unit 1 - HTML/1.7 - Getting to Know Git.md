# 1.7 - Getting to Know Git

The `git` system is extremely complex. **It is very easy to make a mistake, so don't run any `git` commands you aren't 100% certain of!**

### Git runs locally on your development system

**Git** creates a `repository` (think of it like a hidden database) in your current folder.
```SH
# Initialize an empty repository
user@system:/my/folder/$ git init
```

When you run a `git` command, you are working on your development system. Everything gets stored in a hidden `.git` folder _on that computer_ (don't mess with that folder!). If you want to back that up in case of disaster, you use a _remote repository_ like **GitHub**, **Gitea**, **GitLab**, **Bitbucket**, etc...

The point of `git` is to create _described_ snapshots in time.
- Fix a bug? Take a snapshot with a description.
- Added a feature? Take a snapshot describing the added feature. 

This allows you to go back in time if you need to find deleted code, realize a mistake, or actually revert to a saved version that worked better.

### Git Commands: `Clone`, `fetch`, `pull`, `push`, `stage`, `commit`, `status`

There are **_so many git commands_** but here are the important ones:

```BASH
# To "stage" a single file:
user@system:/my/folder/$ git add [filename]

# To "stage" all new / changed / deleted files:
user@system:/my/folder/$ git add .

# To create a local snapshot in time (aka "commit")
user@system:/my/folder/$ git commit -m "A useful message"

# If you have a remote repo, upload all commits to your remote:
user@system:/my/folder/$ git push

# To download a copy of a remote repository from the Internet:
user@system:/my/folder/$ git clone [url]

# To see if there are any changes on the remote repository:
user@system:/my/folder/$ git fetch

# To download an apply changes from the remote repo:
user@system:/my/folder/$ git pull

# To get the status of your repo:
user@system:/my/folder/$ git status
```

### You can create "alternate universes"

Say you have a perfectly working version of your code and you don't want to break it with your next "feature". Git allows you to create a `branch` that is a _copy_ of your current code. You can then work in this _alternate_ `branch` without destroying your `main` branch. If you like your changes, you can `merge` them back into your main branch.

**Note:** branches and merging is a bit advanced. Don't try it until you're ready!
